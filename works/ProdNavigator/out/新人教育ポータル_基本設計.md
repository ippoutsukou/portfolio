# 新人教育ポータル 基本設計

## 1. 目的

- 新人が「商品」「工場」「仕向け先」の関係を短時間で理解できる静的Webアプリを構築する。
- 業務システムではなく、教育用の参照・学習ポータルとして設計する。
- 初版は閲覧中心とし、データ更新は JSON 編集で対応する。

## 2. 対象範囲

### 対象

- トップ、一覧、詳細、関係マップ、よくある間違い、学習モード、用語集/FAQ
- 商品 40 件前後、工場 10 件前後、仕向け先 30 件前後の閲覧
- キーワード検索、カテゴリ絞り込み、相互リンク遷移

### 非対象

- ログイン/権限管理
- Web 上でのデータ編集
- サーバーサイド処理、DB、承認フロー
- AI チャット、外部サービス連携

## 3. 成功条件

- 新人がトップから 3 クリック以内で目的の詳細へ到達できる。
- 商品詳細から関連工場・仕向け先へ相互遷移できる。
- 学習モードで優先学習対象と確認クイズを提供できる。
- JSON 更新だけでコンテンツ差し替えができる。

## 4. システム構成

### 方式

- 静的 Web アプリ
- フロントエンドのみで動作
- データソースは `data/*.json`

### 推奨技術

| 項目 | 選定 |
|------|------|
| UI | HTML / CSS / JavaScript |
| 構成 | Vanilla JS ベースの SPA 風構成 |
| 検索 | 自前フィルタで開始し、必要時に Fuse.js を検討 |
| 関係マップ | 初版は SVG/CSS ベース、複雑化したら Cytoscape.js を検討 |
| 配布 | GitHub Pages またはローカル配布 |

### 技術判断

- 初版は依存を極力増やさず、保守しやすさを優先する。
- 利用者と教育担当者の運用負荷を下げるため、ビルド不要構成を第一候補とする。
- 関係マップは件数が増えても破綻しないよう、描画とデータ整形を分離する。

## 5. 画面設計

### 5.1 画面一覧

| 画面ID | 画面名 | 主目的 |
|------|------|------|
| S-01 | トップ | 全体案内、導線、統計表示 |
| S-02 | 商品一覧 | 商品を探す |
| S-03 | 商品詳細 | 商品の理解を深める |
| S-04 | 工場一覧 | 工場を探す |
| S-05 | 工場詳細 | 工場の担当範囲を確認する |
| S-06 | 仕向け先一覧 | 仕向け先を探す |
| S-07 | 仕向け先詳細 | 仕向け先条件を確認する |
| S-08 | 関係マップ | つながりを視覚把握する |
| S-09 | よくある間違い | 取り違えを防ぐ |
| S-10 | 学習モード | 優先学習と理解度確認 |
| S-11 | 用語集/FAQ | 業務用語の補助理解 |

### 5.2 共通UI部品

- ヘッダー: グローバルナビ、検索導線
- サイド/補助ナビ: 現在地、関連画面リンク
- カード一覧: 商品・工場・仕向け先
- テーブル一覧: 比較・並び替え向け
- タグ/チップ: カテゴリ、注意、優先度
- 相互リンクバッジ: 関連商品、関連工場、関連仕向け先
- アコーディオン: FAQ、用語説明
- ステッパー: 学習モード進捗

### 5.3 画面遷移方針

- 一覧 -> 詳細 の基本導線を持つ。
- 詳細画面同士は相互リンクで横断可能にする。
- 関係マップ、よくある間違い、学習モードからも各詳細へ遷移可能にする。

## 6. データ設計

### 6.1 エンティティ

| エンティティ | 役割 | 主キー |
|------|------|------|
| products | 商品情報 | `id` |
| factories | 工場情報 | `id` |
| destinations | 仕向け先情報 | `id` |
| relations | 商品-工場-仕向け先の関係 | `productId + factoryId + destinationId` |
| mistakes | よくある間違い | `id` |
| quiz | クイズ | `id` |
| glossary | 用語集 | `id` |
| faq | FAQ | `id` |

### 6.2 データモデリング方針

- 商品・工場・仕向け先のマスタは正規化して分離する。
- 関係は `relations.json` で持ち、画面側で逆引き可能な形に整形する。
- 一覧表示に必要な派生項目は描画前に集約する。
- データ欠損は `null` または空配列で扱い、推測値を入れない。

### 6.3 最低限必要な追加スキーマ

初版実装を安定させるため、以下の補助項目を追加推奨とする。

| ファイル | 項目 | 用途 |
|------|------|------|
| `mistakes.json` | `id`, `title`, `category`, `relatedProductIds`, `relatedFactoryIds`, `relatedDestinationIds`, `description`, `prevention` | 事例管理 |
| `glossary.json` | `id`, `term`, `reading`, `category`, `description` | 並び順、検索 |
| `faq.json` | `id`, `category`, `question`, `answer` | FAQ 表示 |

## 8. 処理フロー

### 8.1 初期表示

1. アプリ起動
2. 各 JSON を読込
3. 必須項目を簡易検証
4. 関係データを整形
5. 件数サマリを生成
6. 初期画面を描画

### 8.2 一覧検索

1. ユーザーがキーワード入力
2. 対象一覧データを取得
3. キーワード、カテゴリ、タグ条件で絞り込み
4. ソート適用
5. 件数と結果を再描画

### 8.3 詳細表示

1. URL ハッシュまたは選択イベントから ID を取得
2. 該当マスタを取得
3. `relations` から関連工場・仕向け先を逆引き
4. 注意点、ミス、類似商品を組み合わせて描画

### 8.4 学習モード

1. 優先度付き商品と主要拠点を抽出
2. 推奨順を段階表示
3. 閲覧済み状態を LocalStorage に保存
4. クイズ出題、採点、解説表示

## 9. 例外設計

| ケース | 対応 |
|------|------|
| JSON 読込失敗 | エラーメッセージ表示、対象ファイル名を案内 |
| 必須キー欠損 | 該当レコードをスキップし、警告表示 |
| 関係先 ID 不整合 | 関係データのみ除外し、画面は継続表示 |
| 検索結果 0 件 | 条件クリア導線を表示 |
| LocalStorage 使用不可 | 進捗保存なしで継続利用 |

## 10. 非機能設計

### 性能

- JSON 読込後はメモリ上キャッシュを使い、再読込を避ける。
- 一覧はクライアントサイドで即時フィルタする。

### 保守性

- UI ロジックとデータ整形を分離する。
- JSON スキーマの説明を README に明記する。

### アクセシビリティ

- 見出し構造を守る。
- ボタン、リンク、入力欄に適切なラベルを付ける。
- 色だけで状態を表現しない。

## 11. リスクと先回り対応

| リスク | 対応 |
|------|------|
| Excel 由来の表記ゆれ | 取込前にマスタ整理ルールを定義する |
| 関係マップが過密になる | 起点選択と段階表示を前提にする |
| 教育担当者が JSON 更新でミスする | サンプル、項目説明、簡易バリデーションを用意する |
| 初版の画面数が多い | MVP はトップ、一覧、詳細、学習モードを優先する |

## 12. MVP 優先順位

1. データ整備ルール確定
2. トップ
3. 商品一覧/詳細
4. 工場一覧/詳細
5. 仕向け先一覧/詳細
6. 共通検索
7. 学習モード簡易版
8. よくある間違い
9. 用語集/FAQ
10. 関係マップ
